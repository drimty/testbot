# Bug & Feature Request Tracker Bot

Telegram-бот для приёма багов и запросов фич из закрытой группы. Хранит заявки
в SQLite на вашем VPS, даёт админу команды для просмотра и смены статуса,
и автоматически отвечает пользователю в группе при изменении статуса.

## Файлы в репозитории

- `bot.py` — код бота
- `requirements.txt` — Python-зависимости
- `VERSION` — номер версии (для информации, не обязателен для обновлений)
- `.env.example` — пример настроек (реальный `.env` создаётся при установке)
- `roles.example.json` — пример файла ролей/серверов для `/servers` (реальный `roles.json` создаётся на сервере вручную и в git не хранится)
- `install.sh` — установка и обновление на Ubuntu
- `.gitignore` — чтобы `.env`, `roles.json` и файлы базы (`tickets.db`, а также WAL-файлы `tickets.db-wal` / `tickets.db-shm`) случайно не попали в git

## 1. Подготовка

### 1.1 Создайте бота
В Telegram напишите **@BotFather** → `/newbot` → следуйте инструкциям →
сохраните токен вида `123456789:AA...`.

### 1.2 Узнайте свой Telegram ID
Напишите **@userinfobot** — он пришлёт ваш числовой ID. Это будет `ADMIN_IDS`.
Если админов несколько — перечислите ID через запятую.

### 1.3 Загрузите файлы на GitHub
Создайте репозиторий (публичный — это упрощает скачивание через `raw.githubusercontent.com`
и не содержит секретов, так как токен хранится только в `.env`, который в репозиторий не попадает).
Залейте туда: `bot.py`, `requirements.txt`, `VERSION`, `.gitignore` (и `install.sh` тоже, для истории/обновлений).

### 1.4 Проверьте REPO_RAW_URL в install.sh
В начале `install.sh` задан адрес raw-файлов репозитория:

```bash
REPO_RAW_URL="https://raw.githubusercontent.com/drimty/testbot/main"
```

Для этого репозитория он уже настроен. Если вы делаете форк — замените
`drimty/testbot` на свой `логин/репозиторий` и залейте обновлённый `install.sh`
в тот же репозиторий.

## 2. Установка на VPS (Ubuntu)

Подключитесь к серверу по SSH и выполните:

```bash
wget https://raw.githubusercontent.com/drimty/testbot/main/install.sh
sudo bash install.sh
```

Скрипт спросит:
- **токен бота** (из шага 1.1)
- **ID администраторов** (из шага 1.2)

После этого он сам:
1. поставит Python и зависимости,
2. создаст отдельного системного пользователя `bugtracker` (бот работает не под root),
3. настроит systemd-сервис `bugtracker-bot` с автозапуском и автоперезапуском при сбое,
4. запустит бота.

Проверить, что всё работает:
```bash
systemctl status bugtracker-bot
journalctl -u bugtracker-bot -f
```

## 3. Добавление бота в приватную группу

1. Откройте группу в Telegram → участники → «Добавить участника».
2. Найдите бота по username (тот, что задавали в BotFather) и добавьте.
3. Никаких дополнительных настроек privacy в BotFather делать **не нужно** —
   бот реагирует только на команды (`/bug`, `/feature` и т.д.), а команды
   Telegram доставляет боту всегда, независимо от режима приватности группы.

## 4. Как пользоваться (обычный участник группы)

В группе:
```
/bug после обновления профиля пропадает аватарка
```
Бот ответит (reply на ваше сообщение): «✅ Баг принят, заявка #1».

```
/feature хотелось бы тёмную тему в приложении
```
→ «✅ Запрос фичи принят, заявка #2»

Можно и в два шага (удобно при выборе из меню «/»): отправьте просто `/bug`
(или `/feature`) без текста — бот попросит описание, и **следующим сообщением
в ответ** на его запрос вы отправляете текст заявки. Отмена ввода — `/skip`.

Отменить свою заявку (только пока её не взяли в работу, т.е. статус «🆕 Новая»):
```
/cancel 1
```
Бот переведёт заявку в статус «🗑 Отменена» и уведомит администраторов.
Отменить можно только собственную заявку.

Открыть личный чат с ботом (кнопка со ссылкой):
```
/bot
```

Зарегистрироваться (можно в любой теме) — бот отправит администратору ваше имя
и ID, а сообщение с командой сразу удалит:
```
/signin
```

Узнать свою роль (только в личке):
```
/myrole
```

**Меню команд:** при вводе `/` Telegram показывает список команд — свой для
личного чата и свой для группы (одинаковый во всех темах; отдельного меню на
тему Telegram не поддерживает).

**Чтобы не засорять группу**, «многословные» команды — `/help`, `/status НОМЕР`
и `/mytickets` — работают только в **личном чате** с ботом. Если написать их в
группе, бот коротко (без звука) предложит перейти в личку и через некоторое
время сотрёт подсказку. Создание заявок (`/bug`, `/feature`) остаётся в группе.

## 5. Как пользоваться администратору

**Важно:** боты не могут писать первыми — прежде чем администрировать,
откройте **личный чат с ботом** и один раз нажмите `/start`. Все команды
ниже пишите тоже в личном чате с ботом (не в группе), чтобы не засорять её.

### Тестовый сценарий целиком

1. В группе кто-то (или вы сами) пишет:
   ```
   /bug тестовая заявка, проверка бота
   ```
   Бот отвечает: «✅ Баг принят, заявка #1».

2. В личном чате с ботом смотрите список новых заявок:
   ```
   /list
   ```
   (по умолчанию `/list` показывает статус `new`; можно `/list all`,
   `/list in_progress`, `/list done`, `/list rejected`, `/list cancelled`)

3. Открываете конкретную заявку — появится полный текст и кнопки:
   ```
   /view 1
   ```
   Кнопки: 🆕 Новая / 🔧 В работе / ✅ Готово / 🚫 Отклонено.

4. Нажимаете нужную кнопку (например «🔧 В работе») — бот:
   - обновит статус в базе,
   - обновит это же сообщение в вашем чате,
   - **отправит ответ в группу** (reply на исходное сообщение автора):
     «🔔 Статус вашей заявки #1 изменён: 🔧 В работе».

5. Если нужно оставить текстовый комментарий (например, объяснить решение),
   в личном чате с ботом:
   ```
   /comment 1 Не смогли повторить баг, уточните версию приложения и модель телефона
   ```
   Комментарий появится в группе как reply на исходное сообщение.

Изменить статус можно и без кнопок, командой:
```
/setstatus 1 done
```

Опубликовать в группе закрепляемую панель с кнопкой «Открыть бота» (вызывать
**в группе**, в нужной теме; бот попробует сам закрепить сообщение):
```
/panel
```

Очистить **все** заявки и начать нумерацию заново (например, после тестов).
Требует подтверждения, сообщения в группе не трогает:
```
/reset CONFIRM
```

Список зарегистрированных (через `/signin`) пользователей — их имена и ID
(удобно для заполнения ролей в `roles.json`):
```
/users
```

Ожидающие заявки на вступление в группу (см. раздел 8):
```
/requests
```

Назначить участнику метку-роль (бот должен быть админом группы с правом
«Управление метками»; см. раздел 7):
```
/settag 123456789 Наладчик
```

Узнать текущую версию бота:
```
/version
```

## 6. Снижение флуда в группе (темы форума, эфемерные сообщения)

Эти настройки необязательны — по умолчанию бот ведёт себя как раньше. Задаются
в `/opt/bugtracker-bot/.env` (см. также `.env.example`). После изменения `.env`
перезапустите бота: `sudo systemctl restart bugtracker-bot`.

### Привязка команд к темам форума (topics)

Если группа переведена в режим тем (форум), можно принимать `/bug` и `/feature`
только в отведённых темах:

```
BUG_TOPIC_ID=10
FEATURE_TOPIC_ID=20
```

- Пусто (по умолчанию) — команда работает в любой теме.
- В «неправильной» теме бот коротко (без звука) подскажет, где писать команду.
- Ответы о смене статуса приходят в ту же тему, где создана заявка.

Как узнать ID темы: откройте нужную тему в веб-версии Telegram — число в конце
URL (`.../<chat>/<thread_id>`) и есть ID темы. Либо перешлите сообщение из темы
боту вроде @userinfobot, который покажет `message_thread_id`.

### Эфемерные (самоудаляющиеся) служебные сообщения

```
EPHEMERAL_TTL=60
```

Через указанное число секунд бот удалит служебный «шум» в группе: подсказки,
редиректы в личку, сообщения `/help` и `/bot`. **Заявки `/bug`/`/feature`,
подтверждения о приёме и уведомления о смене статуса не удаляются.**

`0` (по умолчанию) — не удалять. Чтобы бот мог удалять **чужие** сообщения
(например, команду пользователя), сделайте его **администратором группы с правом
«Удалять сообщения»**. Свои сообщения бот удаляет всегда (в пределах 48 часов —
ограничение Telegram).

> Примечание про Privacy Mode: для текущих функций менять его не нужно — бот
> и так получает все команды. Отключать Privacy Mode в @BotFather потребовалось
> бы только для полной автомодерации (удаления вообще всех не-командных
> сообщений в теме), которая в этой версии не реализована.

## 7. Ролевой доступ к серверам (команда /servers)

Команда `/servers` (только в личном чате) показывает пользователю список
серверов **в зависимости от его роли** (например, «Наладчик» / «Метролог»).
Секретные данные хранятся в отдельном файле на сервере и **не попадают в git**.

> Как узнать ID пользователей для `roles.json`: попросите их отправить боту
> `/signin` (можно в любой теме), затем выполните `/users` — бот покажет имена
> и ID всех зарегистрированных. Останется вписать нужные ID в секцию `users`.

### Настройка

1. Скопируйте пример и заполните своими данными:
   ```bash
   sudo cp /opt/bugtracker-bot/roles.example.json /opt/bugtracker-bot/roles.json
   sudo nano /opt/bugtracker-bot/roles.json
   ```
2. Структура файла: секция `users` сопоставляет **Telegram user id → список
   ролей**, секция `roles` задаёт для каждой роли `title` и список серверов
   PostgreSQL (`servers`). У каждого сервера-блока:
   - `title` — заголовок блока;
   - `hosts` — список хостов `{"address", "port"}` (если серверов несколько с
     одинаковыми БД/пользователями, просто добавьте ещё элементы в `hosts`);
   - `db` — имя базы;
   - `accounts` — список `{"username", "password", "desc"}`.
   ```json
   {
     "users": { "111111111": ["naladchik"] },
     "roles": {
       "naladchik": {
         "title": "Наладчик",
         "servers": [
           {
             "title": "Группа серверов 1",
             "hosts": [
               {"address": "150.0.0.1", "port": 5432},
               {"address": "150.0.0.2", "port": 5332}
             ],
             "db": "name_DB",
             "accounts": [
               {"username": "admin",  "password": "passForAdmin",  "desc": "суперпользователь"},
               {"username": "editor", "password": "passForEditor", "desc": "редактор"},
               {"username": "user",   "password": "passForUser",   "desc": "только просмотр"}
             ]
           }
         ]
       }
     }
   }
   ```
   Полный пример — в `roles.example.json`.
3. Права на файл (в нём чувствительные данные, включая пароли):
   ```bash
   sudo chown bugtracker:bugtracker /opt/bugtracker-bot/roles.json
   sudo chmod 600 /opt/bugtracker-bot/roles.json
   ```

Файл **перечитывается на каждый запрос** `/servers`, поэтому после правок
перезапуск бота не нужен. Если у пользователя нет роли (или файл отсутствует),
бот вежливо сообщит, что доступа нет / список недоступен, и запишет причину в лог.

**Копирование по клику:** адрес, порт, имя БД, логин и пароль выводятся
моноширинным шрифтом — в официальных клиентах Telegram тап (клик) по такому
полю копирует именно его в буфер обмена.

### Откуда бот берёт роли

Роли пользователя объединяются из трёх источников:

1. **`roles.json`, секция `users`** (по `user_id`) — работает для всех, основной способ.
2. **`custom_title`** — «должность» администратора группы (только у тех, кто реально в списке админов).
3. **`sender_tag`** — Telegram-метка обычного участника (роль без прав админа).
   Приходит в сообщениях супергруппы; бот запоминает её, когда участник пишет
   в группе (например, командой `/signin` или `/bug`). Метка сопоставляется с
   ролью по её `title` или ключу в `roles.json` (регистронезависимо).

> Нюанс `sender_tag`: чтобы бот узнал метку, участник должен **хотя бы раз
> написать в группе** после её назначения (тег приходит только в сообщении
> супергруппы, в личке его нет). До этого `/servers` роль по метке не увидит.

Назначить метку участнику можно вручную в Telegram или командой бота
`/settag USER_ID метка` (в личном чате). Для этого бот должен быть
**администратором группы с правом «Управление метками» (`can_manage_tags`)**.
Пустая метка (`/settag USER_ID`) снимает её.

## 8. Доступ только участникам группы (заявки на вступление)

По умолчанию бот отвечает всем. Если задать в `.env` ID основной группы —
в **личном чате** бот будет обслуживать только её участников; остальным доступна
лишь команда `/join` (заявка на вступление).

```
GROUP_ID=-1004333980163
```

- ID группы: из ссылки на сообщение `t.me/c/<ЧИСЛО>/...` берётся `<ЧИСЛО>` с
  префиксом `-100` (например, `t.me/c/4333980163/...` → `-1004333980163`).
- Бот должен быть **участником этой группы** (он проверяет членство через API).
- **Администраторы бота** (`ADMIN_IDS`) работают всегда, даже если не в группе —
  это подстраховка от неверного `GROUP_ID`.
- Членство кэшируется на ~60 секунд (только что принятого пользователя бот
  «увидит» в течение минуты).

Как это выглядит:

- Не-участник пишет боту что угодно (или `/start`) → бот отвечает, что доступ
  только для участников, и предлагает `/join`.
- `/join` (можно с комментарием: `/join я из отдела наладки`) → **всем
  администраторам** приходит имя, username и ID заявителя; заявка сохраняется.
- Админ смотрит ожидающих командой `/requests` (в личном чате) и **добавляет
  человека в группу вручную**. Как только он вступил, из `/requests` он пропадёт
  автоматически.

> Бот не добавляет участников сам (у ботов нет такого права без сложной настройки
> инвайт-ссылок) — он лишь собирает заявки и показывает их администратору.

### Секретное слово — вход по ссылке минуя заявку

Можно задать «секрет» и ссылку-приглашение: если пользователь введёт
`/join СЛОВО`, бот сразу пришлёт кнопку со ссылкой на вступление (без ожидания
администратора):

```
GROUP_INVITE_LINK=https://t.me/+ВашаСсылкаПриглашение
JOIN_SECRET=Чубака
```

- Ссылку создайте в настройках группы (Пригласительные ссылки).
- Сравнение слова регистронезависимое (`Чубака` = `чубака`).
- Если секрет не задан, `/join СЛОВО` считается обычным комментарием к заявке.

Сообщение-приглашение для не-участников оформлено в тематическом стиле
(«📜 Голокрон запечатан…») и сопровождается стикером. Стикер задаётся в
`WELCOME_STICKER_ID` (пусто — не отправлять; по умолчанию задан в коде).

## 9. Обновление бота

Когда вы что-то поменяете в `bot.py` и зальёте изменения на GitHub
(желательно также поднять номер в файле `VERSION`, хотя это не обязательно —
скрипт сравнивает содержимое `bot.py` по хэшу), на сервере выполните:

```bash
sudo bash /opt/bugtracker-bot/install.sh
```

Скрипт скачает свежие файлы, сравнит их с установленными и, если есть
изменения — обновит и перезапустит бота. `.env` и база `tickets.db`
никогда не перезаписываются.

### Автоматическая проверка обновлений (опционально)

Чтобы сервер сам проверял обновления, например, раз в сутки:

```bash
sudo crontab -e
```

и добавьте строку:

```
0 4 * * * bash /opt/bugtracker-bot/install.sh >> /var/log/bugtracker-bot-update.log 2>&1
```

## 10. Где хранятся данные и как их бэкапить

Все заявки лежат в `/opt/bugtracker-bot/tickets.db` (SQLite). Рекомендуется
периодически копировать этот файл в безопасное место, например через cron:

```bash
0 3 * * * cp /opt/bugtracker-bot/tickets.db /root/backups/tickets-$(date +\%F).db
```

### Очистить базу и начать заново

Два способа:

- **Командой бота:** в личном чате отправьте `/reset CONFIRM` — удалит все заявки
  и сбросит нумерацию с #1 (сообщения в группе не трогает).
- **На сервере** (полный сброс файла БД):
  ```bash
  sudo systemctl stop bugtracker-bot
  sudo rm -f /opt/bugtracker-bot/tickets.db /opt/bugtracker-bot/tickets.db-wal /opt/bugtracker-bot/tickets.db-shm
  sudo systemctl start bugtracker-bot
  ```
  Бот создаст пустую базу при запуске.

## 11. Диагностика

| Проблема | Что проверить |
|---|---|
| Бот не отвечает | `systemctl status bugtracker-bot` — работает ли сервис |
| Ошибка при запуске | `journalctl -u bugtracker-bot -n 50` — последние строки лога |
| Бот не видит команды в группе | Убедитесь, что бот действительно добавлен как участник группы |
| Админ-команды пишут «⛔ Эта команда доступна только администратору» | Проверьте, что ваш ID верно указан в `ADMIN_IDS` в `/opt/bugtracker-bot/.env` |
| Админ-команды пишут «⛔ Административные команды работают только в личном чате» | Пишите админ-команды (`/list`, `/view`, `/setstatus`, `/comment`) в личном чате с ботом, а не в группе |
| Кнопка «Открыть бота в личке» не появляется / ведёт не туда | Бот должен успешно стартовать (в логах строка `Бот запущен как @username`); ссылка строится из username бота |
| Служебные сообщения не удаляются | Проверьте `EPHEMERAL_TTL>0` в `.env`; для удаления чужих сообщений бот должен быть админом группы с правом «Удалять сообщения»; Telegram не даёт удалять сообщения старше 48 ч |
| `/bug` или `/feature` не принимаются в теме | Сверьте `BUG_TOPIC_ID`/`FEATURE_TOPIC_ID` в `.env` с реальным ID темы; пусто — команда работает в любой теме |
| `/servers` пишет «список недоступен» | Файл `roles.json` отсутствует или невалиден — проверьте путь `ROLES_FILE` и синтаксис JSON; причина будет в логах (`journalctl -u bugtracker-bot`) |
| `/servers` пишет «нет доступа» | Ваш Telegram user id не добавлен в секцию `users` файла `roles.json`, либо указана роль, которой нет в секции `roles` |
| Бот всем отвечает «доступ только для участников» | Проверьте `GROUP_ID` в `.env` (правильный ID, с префиксом `-100`) и что бот добавлен в эту группу; админы из `ADMIN_IDS` работают всегда |
| `/requests` пуст, хотя люди слали `/join` | Заявители уже стали участниками (тогда они убираются из списка), либо `/join` слали при незаданном `GROUP_ID` |
